# Design: AI Auto-Cut Video — Mechanical MVP + Dual Distribution

## Problem Statement

OPC（一人公司）创业者和口播视频创作者面临大量视频剪辑需求。剪辑耗时（每条 1-3 小时）、外包贵（单条 200-500 元）、工具学习门槛高。微信社群中有人直接发招聘找剪辑师，小红书上有大量口播剪辑需求。现有工具（剪映、Descript）解决了部分问题，但仍需大量手动操作，没有实现"素材进，成品出"的全自动体验。

## Demand Evidence

- 微信社群中观察到创作者直接发布招聘剪辑师的需求帖（行为证据：愿意花钱解决）
- 小红书上有大量口播剪辑相关搜索和需求帖
- 创始人自己做过口播，亲身体验录音磕磕绊绊、环境画质差、成品平淡等痛点
- 上周接触到的微商用户，有大量分段素材需要拼接的强需求
- 注意：以上为"有人在花钱/花时间解决这个问题"的信号，尚未验证"愿意为 AI 工具付费"

## Status Quo

目标用户当前采用**混合方案**：
- 自己用剪映手动剪辑（耗时但免费）
- 外包给自由职业剪辑师（每条 200-500 元）
- 用剪映 AI 字幕等功能部分自动化，但仍需大量手动操作
- 具体的用户工作流、时间/金钱成本**尚未深入调研**——这是需要立即验证的关键盲区

## Target User & Narrowest Wedge

**目标用户画像：**
- 口播日更/周更的内容创作者（知识博主、个人 IP、微商）
- 有大量分段素材需要拼接
- 技术水平偏小白，不想学复杂工具
- 要求"量大够用"而非"极致精美"

**最小楔子（Narrowest Wedge）：**
上传口播原始素材 → AI 自动去口误/停顿 → 自动加字幕 → 自动分段拼接 → 导出成品视频

## Constraints

1. 创始人独立开发，资源有限
2. AutoCut.video 品牌已存在（口播+电商自动剪辑），需要品牌差异化
3. 剪映 AI 功能持续增强，竞争窗口可能缩小
4. AI 推理成本（语音识别+处理）需要控制在可盈利范围内
5. 用户审美个性化程度高，创意层自动化需要长期演进

## Premises

1. **口播视频剪辑市场足够大且增长足够快** — AGREED
2. **MVP 先解决机械层（去口误/字幕/分段拼接）** — AGREED（创意层个性化太强，不适合 MVP）
3. **多智能体架构先做机械自动化** — REVISED（创意层后续通过用户自定义风格/模板引入）
4. **用户愿意为 AI 自动剪辑付费** — AGREED（但需要验证）
5. **可扩展到课程、电商等场景** — AGREED
6. **双分发策略：GUI + OpenClaw Skill** — NEW（嵌入用户 AI Agent 工作流，扩大 TAM）

## Approaches Considered

### Approach A: Mechanical MVP + Dual Distribution (CHOSEN)
先做去口误/加字幕/多段合并的自动化引擎，MVP 发布为独立 Web App（OpenClaw Skill 作为 v1.1）。
- Effort: M（4-6 周）
- Risk: Low
- Pros: 技术成熟度高、最快触达用户、OpenClaw 获客成本低
- Cons: 功能与剪映重叠、不解决"视频平淡"的核心痛点、ARPU 天花板低

### Approach B: Creative Enhancement Engine + Template Marketplace
在机械层基础上加入风格模板系统（"知识干货风""情绪共鸣风"等），社区可共享/交易模板。
- Effort: L（8-12 周）
- Risk: Medium
- Pros: 解决最痛的问题、模板市场产生网络效应
- Cons: 技术复杂度高、需要先积累优质模板

### Approach C: AI Editing Agent Platform (Agent-as-a-Service)
做 API/SDK，让其他 App 调用"视频剪辑 Agent"能力。
- Effort: M（4-8 周）
- Risk: Medium-High
- Pros: 避开 to-C 竞争、B2B ARPU 高
- Cons: 缺乏直接用户反馈、需要合作伙伴

## Recommended Approach

**Approach A: Mechanical MVP + Dual Distribution**

理由：技术风险最低（Whisper + FFmpeg 即可实现）、验证速度最快、双分发策略（GUI + OpenClaw Skill）已提供差异化切入点。同时为 Approach B（模板市场）和 C（API 平台）奠定技术基础。

**核心处理流水线（Pipeline，非自主决策 Agent）：**

```
Stage 0: 素材合并（可选）
  Input:  多个视频文件（用户可上传 1~N 个片段）
  Output: 按用户排序合并的单个视频
  Tool:   FFmpeg concat demuxer
  说明:   支持拖拽排序，解决微商等用户"多分段素材需要拼到一起"的需求

Stage 1: 语音识别
  Input:  单个视频文件（MP4/MOV/WebM, ≤1080p, ≤500MB）
  Output: 带时间戳的文字稿 JSON [{text, start_ms, end_ms, confidence}]
  Tool:   Whisper API（MVP 阶段用 API，本地部署需 GPU 留到成本优化阶段）
  注意:   MVP 只支持普通话，方言为已知局限

Stage 2: 口误检测与标记
  Input:  文字稿 JSON（使用 Whisper word-level timestamps）
  Output: 标记了口误片段的 JSON [{type, start_ms, end_ms}]
  实现:   Python 规则引擎（正则 + 时间差判断），不依赖 LLM
  MVP 口误规则（3 种）：
    - 重复词/短语（连续 2+ 个相同 token）→ 保留最后一次
    - 填充词（"嗯""啊""那个""就是"）→ 默认删除，用户可配置保留
    - 长停顿（>1.5s，阈值可调）→ 默认缩短至 0.5s
  v1.1 扩展：
    - 错误后重说（"把这个...不对，把那个"）→ 需要 LLM 语义理解，MVP 不做

Stage 3: 用户预览与确认
  Input:  文字稿 + 口误标记
  Output: 用户确认后的最终删除列表
  界面:   文字稿高亮标注（红色=将删除），用户点击可恢复保留
         + 一键试听按钮（播放处理后音频片段）
  说明:   不做可视化视频时间线编辑器，只做文字稿级别的预览

Stage 4: 视频剪切与拼接
  Input:  原始视频 + 确认后的删除列表
  Output: 去除口误后的视频
  策略:   先尝试 -c copy（最近 I-frame 切割）；若切割误差 > 500ms，
         回退到重编码模式（-c:v libx264）
  音频:   crossfade 50ms 避免爆音
  Tool:   FFmpeg

Stage 5: 字幕生成与烧录
  Input:  剪切后视频 + 文字稿
  Output: 带字幕的视频
  MVP 字幕样式：3 套预设（简洁白字、黑底白字、彩色高亮关键词）
  用户可选：字体大小、位置（上/中/下）、是否显示
  Tool:   FFmpeg drawtext / ASS 字幕

Stage 6: 导出
  Input:  最终视频
  Output: MP4 (H.264 + AAC), 保持原始分辨率和比例
  说明:   MVP 只输出原始比例，多比例导出为 v1.1

异常处理：
  - 上传失败 → 前端文件大小/格式校验 + 后端二次校验，超限返回明确错误
  - Whisper 识别率低（confidence < 0.6 的片段 > 30%）→ 告警用户"音频质量较差，建议重录"
  - FFmpeg 处理超时（>10 分钟）→ 终止任务，通知用户
  - 处理中断 → 保留中间产物，支持从断点恢复
```

**技术栈：**
- 语音识别：Whisper large-v3（优先）/ 阿里云语音（中文方言备选）
- 视频处理：FFmpeg
- 后端：Python (FastAPI)
- 前端：Web GUI（React/Next.js）—— 选型理由：独立开发者全栈效率最高，SSR 支持 SEO
- OpenClaw Skill：封装核心引擎为可调用的 AI 技能

**技术 Spike（开发前必须验证）：**
1. 用 Whisper 转录一段真实口播（含口误、语速快、口语化），评估识别准确率
2. 用 FFmpeg 实现去口误 + 无缝拼接，验证音视频同步和过渡质量
3. 测算单条 10 分钟视频的端到端处理时间和 API 成本

**单位经济学粗算：**
```
处理 10 分钟口播视频（500MB 文件）：
  Whisper API:        ~$0.06（$0.006/min）              ≈ ¥0.45
  服务器转码(CPU):     ~$0.02（按 2x realtime）           ≈ ¥0.15
  带宽(上传+下载):     ~500MB × 2 × ¥0.5/GB             ≈ ¥0.50
  存储(24h 缓存):      ~500MB × ¥0.01/GB/天              ≈ ¥0.01
  支付手续费(2.5%):    按 ¥9.9 收费计                     ≈ ¥0.25
  ─────────────────────────────────────────────────────
  总成本/条:                                             ≈ ¥1.36

定价假设:
  按条: ¥9.9/条 → 毛利率 ~86%
  包月: ¥49/月（含 30 条）→ 单条成本 ¥1.36 × 30 = ¥40.8 → 毛利率 ~17%
  包月: ¥99/月（含 30 条）→ 毛利率 ~59%

结论：按条模式毛利健康；包月定价需 ≥¥99 才可持续
注：后期本地部署 Whisper 需 GPU 服务器（¥500-2000/月），适合量大后降本
```

**扩张路径（参考，非 MVP 范围）：**
口播 MVP → 课程/教程 → 带货口播 → 风格模板系统 → 电商产品视频

## Open Questions

## Success Criteria

**时间线起点：用户访谈完成后（Assignment 完成后）开始计算**

1. **+2 周**：技术 Spike 完成，验证去口误+无缝拼接端到端可行
2. **+8 周**：发布可用 MVP（多段合并 + 去口误 + 自动字幕 + 导出）
3. **+10 周**：获得 20 个真实用户试用并收集反馈
4. **+12 周**：验证付费意愿——至少 3 个用户愿意付费
5. 单条视频处理成本 < 用户付费价格的 30%

**MVP 分发优先级：** Web App 优先（v1.0），OpenClaw Skill 作为 v1.1（避免 MVP 阶段双线作战）

## Distribution Plan

**v1.0（MVP）：独立 Web App**
- 用户上传素材，在线处理，下载成品
- 部署在 Railway/Fly.io（需要后端处理能力，Vercel 不适合长时任务）
- 认证：微信扫码登录（目标用户最熟悉的方式）
- 支付：支付宝/微信支付接入

**v1.1：OpenClaw Skill**
- 将核心引擎封装为 OpenClaw 可调用的 AI 技能
- 待调研：OpenClaw Skill 接口规范、用户基础规模

**获客策略：**
- 小红书/抖音发布对比视频（手动剪辑 vs AI 自动剪辑）
- 微信创作者社群精准投放

## Data Privacy

- 用户上传的视频仅用于处理，处理完成后 24 小时内自动删除
- 如使用 Whisper API（OpenAI），音频数据会发送至 OpenAI 服务器；本地部署模式则全部在自有服务器处理
- MVP 阶段需在上传页面明确告知用户数据处理方式
- 不存储用户视频内容用于模型训练

## Video Format Support (MVP)

- 输入格式：MP4, MOV, WebM
- 最大分辨率：1080p（4K 暂不支持，降低处理成本）
- 最大文件大小：500MB（约 30 分钟 1080p 视频）
- 最大时长：30 分钟
- 输出格式：MP4 (H.264 + AAC)

## Dependencies

- Whisper large-v3（API 或本地部署）
- FFmpeg 视频处理
- OpenClaw Skill 开发框架（v1.1 阶段，待调研接口规范）
- 云服务器：CPU 即可（FFmpeg 转码不强制 GPU），预算约 ¥200/月起

## The Assignment

**不要开始写代码。** 你的下一步是：

**本周内找 5 个在微信社群里发过招聘剪辑师的人，一对一聊 15 分钟。** 每个人问三个问题：
1. 你现在每条口播视频从拍完到发出，整个流程花多长时间？
2. 如果有一个 AI 工具能自动去口误、加字幕、拼接分段，几分钟出成品，你愿意每月花多少钱？
3. 你用剪映吗？剪映的 AI 功能哪里满足你了，哪里没满足？

这 5 个对话能告诉你：
- 真实的工作流成本（现在不知道的关键数字）
- 付费意愿的天花板和底线
- 剪映到底哪里没覆盖到（你的真正机会）

**没有这些数据，任何产品设计都是在猜。**

## What I noticed about how you think

- 你说"前两层花时间练习和花钱可以解决，第三点必须招人"——这是一个非常清晰的价值分层判断，说明你在区分"nice-to-have 的效率提升"和"must-have 的能力缺口"。大多数人分不清这两者。
- 你没有被我的"口播是最佳 MVP"分析带跑——你同意切入口播，但坚持 MVP 只做机械层，因为"每个人都有个性化的审美需求"。这说明你在思考用户真实需求，而不是追求技术上的酷炫。
- "从一开始就做成两套，GUI + OpenClaw Skill"——这个双分发思路是你自己提出的，不是我引导的。这是一个产品直觉：你在想分发渠道，而不只是功能。
- 在 Q2（Status Quo）你坦诚说"还没深入了解"——这种诚实比编造答案有价值得多。但这也意味着你最大的风险不是技术，而是对用户的理解深度。The Assignment 就是解决这个问题。
